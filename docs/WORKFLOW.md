# AI Employee — Vault Workflow

## Overview

The `vault/` directory is the central file-based communication layer between the user,
the watcher system, and the AI processing pipeline. Each subfolder represents a distinct
stage in a task's lifecycle. Files move through folders — the folder a file sits in is
its current status.

---

## Folder Reference

### `vault/Inbox/`
**Purpose:** The primary entry point for all new tasks and inputs.

- The user (or an external system) drops raw task files here.
- Files are unprocessed and unvalidated at this stage.
- Nothing in Inbox has been read or acted on yet.
- Think of it as an unopened email tray.

**What goes here:** plain text tasks, instructions, JSON payloads, notes, anything
that requires future AI action.

---

### `vault/Drop/`
**Purpose:** A fast, no-friction dump zone for informal or bulk input.

- Used when the user wants to quickly offload files without worrying about format.
- Lower priority than Inbox — processed after Inbox is clear.
- Files here may require cleanup or reformatting before they can be acted on.
- Think of it as a physical desktop inbox — messy but useful.

**What goes here:** raw notes, copy-pastes, unformatted ideas, bulk uploads.

---

### `vault/Needs_Action/`
**Purpose:** The active work queue. Holds working copies of all tasks currently being processed.

- A working copy is created here the moment an input is accepted from Inbox or Drop.
- The original source file is moved to Archive at the same time — not after completion.
- A task stays in Needs_Action until it is fully and successfully completed.
- If anything fails (API error, network issue, CLI crash, interruption), the file stays here.
  Nothing is written to Done until the task is confirmed complete.
- This folder is actively watched by the watcher. Archive is not.

**What goes here:** working copies of validated tasks, actively in progress or queued.

---

### `vault/Pending_Approval/`
**Purpose:** Holds AI-generated outputs that require user review before finalizing.

- After the AI processes a task, results that need human sign-off land here.
- The user reviews the file and moves it to `Approved/` or `Rejected/`.
- Nothing leaves this folder automatically — human action required.

**What goes here:** AI drafts, proposals, generated content, decisions needing confirmation.

---

### `vault/Approved/`
**Purpose:** Confirmed outputs that have passed human review.

- Files moved here by the user signal that the AI result is accepted.
- Downstream automation can safely act on files in this folder.

**What goes here:** user-approved AI outputs ready for use or further processing.

---

### `vault/Rejected/`
**Purpose:** Outputs the user has declined or flagged as incorrect.

- Files moved here signal that a task result was unsatisfactory.
- Can be used to trigger a retry, a revision, or simply logged for review.

**What goes here:** failed or refused AI outputs.

---

### `vault/Done/`
**Purpose:** Final AI output for successfully completed tasks only.

- A file is written here only after a task has been fully completed with no errors.
- Contains the AI-generated result — not the original source file.
- The original source file lives in Archive, not here.
- If a task failed, was interrupted, or produced no output, nothing is written here.

**What goes here:** final AI results only. Never source files. Never partial outputs.

---

### `vault/Archive/`
**Purpose:** Storage for original source files. Not watched. Not a work queue.

- The original input file is moved here immediately when a task is accepted —
  before any processing begins.
- This happens at intake, not at completion.
- Archive is never watched by the watcher. Moving files here will not trigger processing.
- Reference only — do not move files out of Archive to re-trigger work.
  Re-submit via Inbox or Drop instead.

**What goes here:** original input files, preserved at the moment of intake.

---

### `vault/Plans/`
**Purpose:** Stores structured plans, multi-step task breakdowns, and project outlines.

- Used when a task requires more than one action to complete.
- Plans may reference files in other folders by name or ID.
- Not a work queue — a reference and coordination layer.

**What goes here:** step-by-step plans, checklists, project briefs, task trees.

---

### `vault/Logs/`
**Purpose:** System logs generated by the watcher and AI pipeline.

- All watcher activity, processing events, and errors are written here.
- Not for user-created files — written to by the system only.
- Primary file: `watcher.log`

**What goes here:** auto-generated log files.

---

### `vault/Dashboard/`
**Purpose:** Snapshot summaries of system state for quick human review.

- Contains auto-generated status files (e.g., counts per folder, last run time).
- Intended to give the user a glance at what the system is doing without opening logs.
- Written to by the system; read by the user.

**What goes here:** status snapshots, summary reports, health checks.

---

## Task Lifecycle

### Standard path (Inbox → approval required)

```
User drops a file
        │
        ▼
  vault/Inbox/
        │
        ▼ (watcher accepts it)
        ├──── original file ──────────────► vault/Archive/   (stored immediately, never touched again)
        │
        ▼ (working copy created)
  vault/Needs_Action/                       (task lives here until fully complete)
        │
        ▼ (AI processes it successfully)
  vault/Pending_Approval/                   (result awaiting human review)
        │
      ┌─┴──────────────────┐
      ▼                    ▼
vault/Approved/       vault/Rejected/
      │
      ▼
  vault/Done/                               (final AI result written here — not the source file)
```

---

### Drop path (informal input)

```
vault/Drop/
        │
        ▼ (same intake as Inbox)
        ├──── original file ──────────────► vault/Archive/
        │
        ▼
  vault/Needs_Action/ → ... (same as above)
```

---

### Bypass path (no approval needed)

```
vault/Needs_Action/
        │
        ▼ (AI completes task, no review required)
  vault/Done/                               (final AI result only)
```

Used for automated tasks where human sign-off is not required.

---

### Failure / interruption path

```
vault/Needs_Action/
        │
        ▼ (error: API failure, network issue, crash, timeout, etc.)
  vault/Needs_Action/                       (file stays — nothing written to Done)
```

The task is retried on the next watcher cycle. Done is never written to on a failed run.

---

## Rules

- One file = one task unit. Do not bundle multiple tasks in one file.
- File names should be descriptive and timestamped where possible.
  Example: `2026-03-25_summarize-report.txt`
- The original source file goes to Archive at intake — not at completion.
- Done contains only final AI output. Never source files. Never partial results.
- A task stays in Needs_Action until it succeeds. Failure keeps it there.
- Archive is not watched. Do not place files there to trigger work — use Inbox or Drop.
- `Logs/` and `Dashboard/` are system-owned. Do not place manual files there.
- When in doubt about where to put a file, use `Inbox/`.
